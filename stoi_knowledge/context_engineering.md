# Context Engineering 知识库

## 核心原则

LLM 的注意力分布不均匀：开头和结尾信息被充分利用，中间信息常被忽视（Lost in the Middle 现象）。即使 token 被付费，被模型忽视的 token 就是浪费。Context engineering 的目标是用有限的 token 量最大化信息利用率。

## 问题模式

### 模式1：中段迷失（Lost in the Middle）

**症状：**
- 多轮对话中，越往后，早期轮的结论被遗忘
- 同样的信息，放在对话开头时有效，放在中间时模型视而不见
- 上下文长度超过 50K tokens 后，质量明显下降

**根因：**
学术研究（Liu et al., TACL 2023）证实 LLM 在长上下文中的性能呈 U 形曲线。模型倾向于关注开头的指令和结尾的问题，中间的信息被"淹没"。在多轮对话中，早期轮的信息逐步后移到上下文中间，其信息利用率随之下降。

**修复：**
1. **重要信息最前置**：系统提示 → 关键背景 → 少量示例（开头不变）
2. **历史摘要而非全文**：第 5 轮后，将轮 1-4 压缩成结构化摘要（保留关键决策点），保持轮 5+ 完整文本
3. **周期性打包**：每 5-8 轮对话，做一次"检查点"总结，后续引用总结而非原始轮

**预期收益：**
- 多轮对话的平均响应质量提升 15-25%
- Token 有效利用率从 60% 提升至 85%+
- 对话可扩展性：16+ 轮不产生明显质量衰减

**参考：**
"Lost in the Middle" 论文；LongLLMLingua (Microsoft, ACL 2024) 在处理长上下文中的 Lost-in-Middle 问题时实现 21.4% 性能提升

### 模式2：上下文膨胀（Context Bloat）

**症状：**
- 每轮新增的上下文往往包含重复、冗余或边缘相关信息
- 完整对话历史逐轮增长，但对后续问题的实际贡献度递减
- 4000+ token 的上下文中，30-40% 的内容可被安全删除而不损失质量

**根因：**
开发者习惯性地将所有过去的交互全量追加，而非进行质量筛选。这源于对 LLM 信息处理能力的误判：开发者假设更多上下文 = 更好的结果，但实际上超长上下文导致成本高昂、延迟增加、质量反而下降。

**修复：**
1. **向量检索而非全量**：用嵌入模型为历史对话建索引，仅检索与当前问题相关的过去交互（top-k 检索）
2. **逐层压缩**：
   - 第 1 层：去除格式装饰符（markdown、过度空行）
   - 第 2 层：合并相似的对话轮（如用户多次询问同一概念，保留最完整的那一个）
   - 第 3 层：抽象化（"用户询问了关于错误处理的 3 个问题" 代替 3 个完整问题）
3. **质量评分截断**：为每条历史记录评分（与当前任务的相关性），仅保留分数 > 0.5 的内容

**预期收益：**
- 平均 token 使用量下降 25-40%
- 成本下降 20-35%，延迟下降 15-25%
- 模型质量指标：无损或提升（因减少噪音）

**参考：**
LLMLingua 系列论文（EMNLP 2023 / ACL 2024）：20 倍压缩率下 RAG 性能维持高位；ProCut（2025）：通过语义单元归因实现 78% 生产环保压缩且性能不降

### 模式3：示例过多或低质

**症状：**
- Few-shot 示例数从 3 个增加到 10 个，但模型输出质量没有相应提升
- 示例中包含边界情况或反面例子，反而混淆模型
- 不同的示例示范不同的风格，模型倾向于平均而非学习最佳风格

**根因：**
示例的质量远比数量重要。一个低质示例的负面影响可能抵消 5 个高质示例的正面效果。此外，多示例时模型会做某种"平均"学习而非精确模仿最优示例。

**修复：**
1. **精选示例**：从同类问题的成功案例中选 2-3 个最佳示范，舍弃平庸示例
2. **示例同质性**：所有示例应遵循相同的风格、格式、难度水平
3. **避免反面示例**：除非明确要求"不要像这样做"，否则不要展示错误示例（反而会被模仿）
4. **示例多样性折衷**：保持 2-3 个示例的多样性（覆盖不同的问题类型），而非 10 个相似示例

**预期收益：**
- token 使用量下降 20-30%（示例变少）
- 输出一致性提升 10-20%（示例质量更高）
- ROI：每减少 1 个低质示例，节省约 200-500 tokens

**参考：**
in-context learning 研究（Atwood et al., 2024）：5-shot 中，示例质量排名比数量排名影响更大

### 模式4：格式冗余

**症状：**
- System prompt 中大量使用 Markdown 装饰符（### 标题、**粗体**、> 引用）
- XML 标签的缩进和格式过度
- JSON 示例的空格和换行未被优化

**根因：**
虽然 Markdown 对人类可读性有帮助，但对 LLM 处理而言是纯粹的冗余。LLM 能理解内容而不需要视觉格式。类似地，未压缩的 JSON（带缩进）相比紧凑 JSON 多消耗 30-50% 的 token。

**修复：**
1. **去 Markdown**：系统提示使用纯文本，仅用简单分界符（`---`, `==`, 数字列表）
2. **JSON 压缩**：用 minified JSON（单行、无空格），但在示例中保持可读格式让模型学习
3. **YAML vs JSON**：对某些结构化提示，YAML 比 JSON 节省 10-15% token（冒号、引号更少）

示例对比：
```
# 之前（Markdown，冗余）
### System Instructions
You are a helpful assistant.
- **Point 1**: Do this
- **Point 2**: Do that
> Important: Always validate input

# 之后（纯文本，简洁）
System Instructions:
You are a helpful assistant.
- Point 1: Do this
- Point 2: Do that
Important: Always validate input
```

**预期收益：**
- System prompt 的 token 使用量下降 15-25%
- 多轮对话中，格式优化带来的累积节省：50-150 tokens/轮

**参考：**
Anthropic 缓存优化案例；ProCut 论文中的 "token efficiency" 部分

### 模式5：检索无关上下文

**症状：**
- RAG 系统中，检索的文档段落往往包含与查询无关的冗长介绍或元数据
- 相关性评分 > 0.8，但检索内容中真正有用的部分仅占 30-40%
- 不同的查询检索出相同的冗长段落，浪费重复的 token

**根因：**
向量相似度计算是全文本的，高分段落可能在某个局部高度相关，但包含大量低相关性的内容。例如，一个关于"数据库连接"的段落可能开头是 5 段关于"一般最佳实践"的冗长介绍，真正相关的 2 段在中间。

**修复：**
1. **文档预处理**：将长文档分割成更小的、语义独立的块（250-400 tokens/块，而非传统的 512/1024）
2. **后处理过滤**：检索后，对每个段落做局部相关性评分，剪裁掉低分头尾部分
3. **元数据剥离**：删除不必要的元数据（如"章节编号"、"版权信息"）
4. **上下文压缩**：用 LLMLingua 或类似工具对检索段落做通道压缩

**预期收益：**
- 平均检索内容的 token 数下降 30-50%
- RAG 系统的检索质量（F1 分数）无降或提升（冗余内容被移除）
- 成本下降 20-30%

**参考：**
Retrieval 系统最佳实践（Anthropic、LangChain 文档）；LLMLingua 的 "Compressing Retrieved Context" 章节

## 诊断问题

**当 STOI 检测到以下模式时，应标记为 Context Engineering 问题：**

1. **Pattern：上下文长度 > 8000 tokens，但 Lost-in-Middle 质量评分 < 0.7**
   → 建议：压缩历史或重新组织顺序

2. **Pattern：过去 10 轮对话累积的上下文中，检索出的相关片段仅占 < 40%**
   → 建议：启用向量检索而非全量历史

3. **Pattern：System prompt 的 token 计数 > 总 token 的 15%，且包含 Markdown 符号**
   → 建议：格式优化

4. **Pattern：Same 文档段落在多个查询中被重复检索**
   → 建议：启用缓存或内容重组

## 改进建议模板

**场景：客服 RAG 系统每个查询都检索 5000+ tokens 但有效信息仅 30%**

### 当前问题：
```
Query: "如何重置密码？"
Retrieved: "关于账户安全的完整指南（包含 20 个章节）"
Relevant: 第 15 章的 3 段（约 1500 tokens）
Waste: 3500 tokens 的无关内容
```

### 改进方案：
1. **文档重构**：按主题分割知识库，"重置密码" 单独为一个 800-token 文档
2. **检索后处理**：
   ```python
   # 检索后过滤
   for chunk in retrieved_chunks:
     relevant_spans = extract_relevant_sentences(chunk, query)
     if sum(token_count(span) for span in relevant_spans) > 100:
       include(relevant_spans)
   ```
3. **启用检索缓存**：常见问题（如"密码重置"、"账户确认"）的检索结果缓存 24 小时

### 预期结果：
```
Query: "如何重置密码？"
Retrieved: 800 tokens（精确匹配文档）
Used: 750 tokens（无冗余）
Saved: 4250 tokens → 85% 节省
Cost: $0.10 → $0.015（降低 85%）
```

### 验证指标：
- ✓ 检索内容的相关性评分 > 0.9
- ✓ 实际使用的 token 占检索 token 的 > 80%
- ✓ 查询延迟 < 2s（快 50%）
