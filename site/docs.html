STOI 的三分层模型

我们把一次 Claude Code 会话的效率，拆成三个可观测、可干预的维度。

第一层：Cache Efficiency（缓存效率）

Anthropic 的 prompt caching 让前缀一致的请求只付 10% 的价格。这意味着：

  • System Prompt 越稳定，命中率越高。
  • Tool result 里动态注入的时间戳、绝对路径、UUID，是 cache miss 的常见元凶。
  • 第一轮建立缓存的成本 unavoidable，但从第二轮开始，每一次 miss 都是白花钱。

第二层：Feedback Validity（输出有效性）

光是 cache 命中还不够——如果模型输出了长篇大论，用户却没有采纳（没有修改文件、没有执行建议、只是忽略了），这些 token 也是事实上的浪费。STOI 用规则 + 可选 LLM 判断，来标注每一轮 assistant 输出的有效性。

  • valid：用户明显采纳了输出（执行了命令、编辑了文件）。
  • invalid：用户没有动作，或明确表达不满。
  • partial：部分采纳，或需要进一步追问。

第三层：Cost Breakdown（成本拆解）

把 cache 命中/未命中的价格差异、有效/无效输出的占比，换算成实际美元。你会看到：

  • 实际花费 vs 没有缓存时的花费
  • 无效输出造成的浪费金额
  • 长 session 中 context 膨胀带来的边际成本

含屎量等级

含屎量越低越好。等级只是一个便于快速理解的标签。

等级 emoji 阈值 含义
CLEAN ✅ 0% ~ 30% 缓存稳定，浪费可控。
MILD_SHIT 🟡 30% ~ 50% 有优化空间，关注一下动态字段。
SHIT_OVERFLOW 🟠 50% ~ 75% 大量 token 没命中缓存，需要排查 System Prompt。
DEEP_SHIT 💩 75% ~ 100% 成本黑洞，建议立即拆分 session 或重写 Prompt。

洛小山四层优化方法论

在链条分析（chain analysis）中，我们进一步把浪费拆成四层，逐层定位、逐层给出可执行建议。

L1 语法层

JSON 缩进、Markdown 装饰、过度换行等格式 token。建议：工具调用参数使用最小化 JSON，去掉不必要的空格和换行。

L2 语义层

Tool result 内容冗余、重复的 context、过大的文件读取结果。建议：给 Claude Code 添加指令——"grep 结果超过 50 行时只返回文件名"。

L3 架构层

Cache miss 的根因：动态字段注入（时间戳、UUID、PID、绝对路径）。建议：在 CLAUDE.md 中明确要求过滤掉这些动态内容。

L4 输出层

Yapping、不必要的总结、过度解释。建议：在 CLAUDE.md 中加入——"完成任务后不要总结已做的事，直接等待下一个指令"。
