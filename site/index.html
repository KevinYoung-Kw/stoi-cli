一个刻薄但可靠的审计师

在这个 AI 狂飙的时代，用得多 ≠ 产得好。我们每天都在和 Claude Code 对话，但很少有人停下来问自己：刚才那一轮，有多少 token 是被浪费的？

STOI 的诞生，就是想让这种「无意识烧钱」变得可见。我们不只告诉你"含屎量 60%"，还会像一位资深工程师一样，指出具体哪一行 System Prompt 在破坏缓存、哪一段 Tool Result 在膨胀上下文、哪一句输出属于无意义的 Yapping。

我们的信条：只给可执行的建议，不说正确的废话。

你在无意识烧钱的 3 个信号

如果你在使用 Claude Code 时遇到过以下情况，你的 token 效率大概率正在「含屎」。

0%

多轮对话中 cache_read_input_tokens 始终为 0。这意味着 System Prompt 或 Tools 中存在动态字段（时间戳 / UUID / 绝对路径），导致 Anthropic 的精确前缀匹配完全失效。

40%

4000+ token 的上下文中，约 30–40% 的内容可被安全删除而不损失质量。开发者习惯性地将完整历史全量追加，而非筛选与当前问题相关的片段。

↑ 线性增长

成本随对话轮数持续上升，没有出现平台期。通常是因为没有使用 /compact 压缩历史，也没有采用滑动窗口策略。提示缓存的写入成本是正常价格的 1.25 倍，如果每轮都重新写，省下来的钱会被消耗殆尽。

STOI 的三分层内核

我们不会给你一个模糊的好/坏评分。STOI 将会话效率拆成三个可观测、可干预的维度，逐层定位根因。

L1 Cache Efficiency

前缀稳定性决定 KV Cache 命中率。命中一次，读取价格仅为正常的 10%。STOI 会帮你找出破坏精确前缀匹配的动态字段。

L2 Feedback Validity

输出再长，如果用户没有采纳（没有改文件、没有执行命令），就是无效 token。我们用规则 + 可选 LLM 判断标注每一轮的有效性。

L3 Cost Breakdown

将 cache 命中/未命中的价格差异、有效/无效输出占比，换算成实际美元。你会看到自己的"真实成本"和"若无缓存时的成本"。

STOI 四层浪费定位法

在链条分析（chain analysis）中，我们把一次对话里的 token 浪费拆成四个层级，逐层给出可执行建议。

L1 输入层

JSON 里多余的缩进、Markdown 装饰符、无意义的换行。建议：使用最小化 JSON，去掉不必要的格式字符。

L2 结果层

重复的 context、过长的 grep 结果。建议：在 CLAUDE.md 中加入规则——"grep 超过 50 行时只返回文件名"。

L3 结构层

时间戳、UUID、PID、绝对路径导致 cache miss。建议：将这些动态内容移到 user message，保持 system prompt 静态。

L4 输出层

Yapping、过度解释、冗余总结。建议：在 CLAUDE.md 中加入——"完成任务后不要总结已做的事"。

Before / After

一个典型的 Claude Code 会话，在 STOI 介入前后的对比。

Before · 无优化

System prompt 中注入当前时间戳和进程 ID，导致每一轮 cache 完全失效。

对话历史无限追加，第 12 轮上下文膨胀到 8000+ tokens。

模型每轮输出都带有一段"总结已完成操作"的废话。

After · 有 STOI

移除 system prompt 中的动态字段，cache 命中率从 0% 提升到 90%+。

在第 8 轮执行 /compact，后续轮次始终保持在 3500 tokens 以内。

在 CLAUDE.md 中禁用了自动总结，平均每轮节省 200–400 tokens。

关键洞察： 缓存失效往往不是因为上下文变长，而是因为 system prompt、tools 或 messages 的细微变化导致精确前缀匹配失败。一次 cache miss 可能导致 25–50% 的成本增加。
